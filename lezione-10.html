L10 · Classificatori naive Bayes

Dato che un supereroe ha gli occhi neri, è più probabile che sia Marvel o DC? E se in più ha anche i capelli neri? Il teorema di Bayes fornisce la cornice probabilistica per rispondere; l'ipotesi «naive» (di indipendenza fra le feature) la trasforma in un algoritmo di classificazione veloce, semplice e sorprendentemente efficace.

≈ 45–60 minuti 8 quiz 4 esercizi numpy, pandas, scikit-learn

1. Richiamo: teorema di Bayes

Per ogni coppia di eventi \(A, B\) con \(\mathrm P(B) > 0\):

\[ \mathrm P(A \mid B) = \frac{\mathrm P(B \mid A)\, \mathrm P(A)}{\mathrm P(B)} \]

  • \(\mathrm P(A)\) — prior: ciò che credo prima di osservare;
  • \(\mathrm P(B \mid A)\) — verosimiglianza: quanto è plausibile osservare \(B\) se \(A\) è vero;
  • \(\mathrm P(A \mid B)\) — posterior: ciò che credo dopo aver osservato \(B\);
  • \(\mathrm P(B)\) — evidenza: serve a normalizzare.
Intuizione. Bayes ribalta il condizionamento. Se so come generare i dati a partire dalla causa (\(\mathrm P(B \mid A)\)), posso inferire la causa dai dati (\(\mathrm P(A \mid B)\)), purché conosca la frequenza a priori della causa stessa.

Quale ruolo ha \(\mathrm P(A)\) nel teorema di Bayes?

  • Verosimiglianza dell'osservazione
  • Prior: probabilità della causa prima di osservare
  • Costante di normalizzazione
  • Probabilità che il classificatore sbagli

2. Classificare con una sola feature

Indichiamo con \(M\) = «un supereroe è Marvel» e \(N\) = «un supereroe ha gli occhi neri». Vorremmo stimare \(\mathrm P(M \mid N)\) usando le frequenze osservate nel dataset.

Storm;Marvel Comics;Brown;White
Thor;Marvel Comics;Blue;Blond
Black Widow;Marvel Comics;Green;Auburn
Captain America;Marvel Comics;Blue;Blond
Doctor Strange;Marvel Comics;Grey;Black
Magneto;Marvel Comics;Grey;Black

Daredevil;Marvel Comics;Blue;Auburn
Black Panther;Marvel Comics;Brown;Black
Punisher;Marvel Comics;Blue;Brown
Deadpool;Marvel Comics;Brown;Brown
Batman;DC Comics;Blue;Black

Wonder Woman;DC Comics;Blue;Black
Joker;DC Comics;Green;Green

Catwoman;DC Comics;Green;Black
Flash;DC Comics;Green;Auburn
Cyborg;DC Comics;Brown;Black
Penguin;DC Comics;Blue;Black
Riddler;DC Comics;Blue;Auburn
"""
import pandas as pd, io

heroes = pd.read_csv(io.StringIO(HEROES_CSV), sep=';', index_col='Name')

# Eventi
marvel = heroes['Publisher'] == 'Marvel Comics'
black_eyes = heroes['Eye color'] == 'Black'
black_eyes_or_dark = heroes['Eye color'].isin(['Brown', 'Grey'])  # nero non presente: usiamo "scuri"

# Per fedeltà al notebook chiamiamo "occhi neri" tutto ciò che non è Blue/Green
black_eyes = ~heroes['Eye color'].isin(['Blue', 'Green'])

p_marvel = marvel.mean()
p_black  = black_eyes.mean()
p_black_given_marvel = (black_eyes & marvel).sum() / marvel.sum()

print(f"P(M)            = {p_marvel:.3f}")
print(f"P(N)            = {p_black:.3f}")
print(f"P(N | M)        = {p_black_given_marvel:.3f}")

p_marvel_given_black = p_black_given_marvel * p_marvel / p_black
print(f"\nP(M | N) = P(N|M) P(M) / P(N) = {p_marvel_given_black:.3f}")

3. L'ipotesi «naive»

Se aggiungiamo una seconda feature \(C\) («colore dei capelli»), il teorema di Bayes diventa:

\[ \mathrm P(M \mid O, C) = \frac{\mathrm P(O, C \mid M)\, \mathrm P(M)}{\mathrm P(O, C)} \]

La verosimiglianza congiunta \(\mathrm P(O, C \mid M)\) è difficile da stimare se le combinazioni sono tante. L'ipotesi naive assume indipendenza condizionata:

\[ \mathrm P(O, C \mid M) \approx \mathrm P(O \mid M)\, \mathrm P(C \mid M) \]

Si stimano così solo \(m + n\) probabilità (somma) anziché \(m \cdot n\) (prodotto). L'ipotesi è quasi sempre falsa, ma in pratica funziona: di qui il nome «ingenuo» (naive).

Name;Publisher;Eye color;Hair color

Spider-Man;Marvel Comics;Hazel;Brown

Storm;Marvel Comics;Brown;White
Thor;Marvel Comics;Blue;Blond
Black Widow;Marvel Comics;Green;Auburn
Captain America;Marvel Comics;Blue;Blond
Doctor Strange;Marvel Comics;Grey;Black
Magneto;Marvel Comics;Grey;Black
Venom;Marvel Comics;Blue;Black
Daredevil;Marvel Comics;Blue;Auburn
Black Panther;Marvel Comics;Brown;Black
Punisher;Marvel Comics;Blue;Brown
Deadpool;Marvel Comics;Brown;Brown
Batman;DC Comics;Blue;Black
Superman;DC Comics;Blue;Black
Wonder Woman;DC Comics;Blue;Black
Joker;DC Comics;Green;Green
Aquaman;DC Comics;Blue;Blond
Catwoman;DC Comics;Green;Black
Flash;DC Comics;Green;Auburn
Cyborg;DC Comics;Brown;Black
Penguin;DC Comics;Blue;Black
Riddler;DC Comics;Blue;Auburn
"""
import pandas as pd, io

heroes = pd.read_csv(io.StringIO(HEROES_CSV), sep=';', index_col='Name')

marvel = heroes['Publisher'] == 'Marvel Comics'
brown_eyes = heroes['Eye color'] == 'Brown'
black_hair = heroes['Hair color'] == 'Black'

p_marvel = marvel.mean()
p_brown_given_marvel = (brown_eyes & marvel).sum() / marvel.sum()
p_blackhair_given_marvel = (black_hair & marvel).sum() / marvel.sum()
p_brown_and_blackhair = (brown_eyes & black_hair).mean()

post = p_brown_given_marvel * p_blackhair_given_marvel * p_marvel / p_brown_and_blackhair
print(f"P(M | occhi=marrone, capelli=neri) ≈ {post:.3f}")

L'ipotesi naive afferma che…

  • Le classi hanno tutte la stessa frequenza a priori.
  • Le feature sono indipendenti dato il valore della classe.
  • Le feature sono numeriche e gaussiane.
  • Il classificatore non sbaglia mai sui dati di training.
La semplificazione cardine è l'indipendenza condizionata: \(\mathrm P(X_1, X_2 \mid Y) \approx \mathrm P(X_1 \mid Y)\, \mathrm P(X_2 \mid Y)\).

4. Più di due classi: regola \(\arg\max\)

Con \(m\) classi \(y_1, \dots, y_m\) e \(n\) feature \(X_1, \dots, X_n\), l'osservazione viene assegnata alla classe

\[ k^* \;=\; \arg\max_k\;\; \mathrm P(Y = y_k) \prod_{i=1}^{n} \mathrm P(X_i = x_i \mid Y = y_k) \]

Il denominatore \(\mathrm P(X_1=x_1, \dots, X_n=x_n)\) è uguale per tutte le classi e quindi può essere ignorato nel confronto.

Attenzione ai prodotti di numeri piccoli. Moltiplicare molte probabilità può portare a underflow numerico. In pratica si lavora in scala logaritmica: \( \log \mathrm P(Y = y_k) + \sum_i \log \mathrm P(X_i \mid Y = y_k) \).

Perché il denominatore \(\mathrm P(X_1, \dots, X_n)\) si può ignorare?

  • Perché vale sempre 1.
  • Perché è sempre uguale al prior.
  • Perché non dipende da \(k\): ininfluente per il \(\arg\max\).
  • Perché lo richiede l'ipotesi naive.

5. GaussianNB con scikit-learn

La libreria sklearn.naive_bayes implementa varie versioni del classificatore in base a come si stima \(\mathrm P(X_i \mid Y)\):

  • GaussianNB — feature continue, gaussiane;
  • MultinomialNB — feature di conteggio (es. parole);
  • BernoulliNB — feature binarie (presenza/assenza);
  • CategoricalNB — feature categoriche.

Tutte le classi seguono la stessa API: model.fit(X, y) e model.predict(X).

import pandas as pd, io
from sklearn.naive_bayes import GaussianNB

heroes = pd.read_csv(io.StringIO(HEROES_CSV), sep=';', index_col='Name')
y = (heroes['Publisher'] == 'Marvel Comics').astype(int)
X = heroes.drop(columns=['Publisher'])

model = GaussianNB().fit(X, y)
pred = model.predict(X)

tp = int(((pred == 1) & (y == 1)).sum())
fn = int(((pred == 0) & (y == 1)).sum())
fp = int(((pred == 1) & (y == 0)).sum())
tn = int(((pred == 0) & (y == 0)).sum())

cm = pd.DataFrame(
    [[tp, fn], [fp, tn]],
    index=['Etichetta positiva (Marvel)', 'Etichetta negativa (DC)'],
    columns=['Predetto positivo', 'Predetto negativo'],
)
print(cm)
print(f"\nAccuracy = (TP + TN) / N = ({tp} + {tn}) / {len(y)} = {(tp + tn) / len(y):.3f}")
Classi sbilanciate. Se la classe positiva è molto più frequente, anche un classificatore «sempre positivo» avrà alta accuracy. Per dataset sbilanciati guardate sempre precision, recall, F1 e la matrice completa, non solo l'accuracy.

7. Esercizi finali

Esercizio 1 — Bayes a mano sui malati

Una malattia ha prevalenza \(\mathrm P(D) = 0{,}01\). Un test ha sensibilità \(\mathrm P(+ \mid D) = 0{,}99\) e specificità \(\mathrm P(- \mid \overline{D}) = 0{,}95\) (cioè \(\mathrm P(+ \mid \overline{D}) = 0{,}05\)). Una persona positiva, qual è la probabilità che sia davvero malata? 

# Tua soluzione qui

p_d = 0.01
p_pos_given_d = 0.99
p_pos_given_no_d = 0.05

p_no_d = 1 - p_d
p_pos = p_pos_given_d * p_d + p_pos_given_no_d * p_no_d
p_d_given_pos = p_pos_given_d * p_d / p_pos

print(f"P(positivo)             = {p_pos:.4f}")
print(f"P(malato | positivo)    = {p_d_given_pos:.4f}")
print("Anche con un test 'molto buono', la rarità della malattia rende dubbio il positivo!")

Esercizio 2 — Naive Bayes a mano (due feature binarie)

Su 100 mail: 60 sono spam, 40 ham. Tra le spam, l'80% contiene la parola «vinci» e il 70% «gratis». Tra le ham, il 10% contiene «vinci» e il 5% «gratis». Una mail contiene entrambe: meglio classificarla come spam?

Confronta \( \mathrm P(\text{spam}) \cdot \mathrm P(\text{vinci}\mid\text{spam}) \cdot \mathrm P(\text{gratis}\mid\text{spam}) \) con il corrispondente prodotto per \(\text{ham}\). Il vincitore non cambia se ometti il denominatore comune. 

# Tua soluzione qui

p_spam = 0.60
p_ham  = 0.40

s_spam = p_spam * 0.80 * 0.70  # P(spam) * P(vinci|spam) * P(gratis|spam)
s_ham  = p_ham  * 0.10 * 0.05  # idem per ham

print(f"score spam = {s_spam:.4f}")
print(f"score ham  = {s_ham:.4f}")
print("→ classifico come:", "SPAM" if s_spam > s_ham else "HAM")

# In probabilità (normalizzate):
tot = s_spam + s_ham
print(f"\nP(spam | mail) ≈ {s_spam/tot:.3f}")
print(f"P(ham  | mail) ≈ {s_ham/tot:.3f}")

Esercizio 3 — Confronto: GaussianNB vs MultinomialNB

Sul dataset cifre load_digits() (8×8 pixel), addestra sia GaussianNB sia MultinomialNB e confronta l'accuracy sui dati di training. Quale funziona meglio? Perché?

I pixel sono interi (0–16): si comportano più come «conteggi» o come valori continui? 

# Tua soluzione qui

from sklearn.datasets import load_digits
from sklearn.naive_bayes import GaussianNB, MultinomialNB

X, y = load_digits(return_X_y=True)

g = GaussianNB().fit(X, y)
m = MultinomialNB().fit(X, y)

print(f"GaussianNB    accuracy: {(g.predict(X) == y).mean():.3f}")
print(f"MultinomialNB accuracy: {(m.predict(X) == y).mean():.3f}")
# Sui pixel 0-16 il MultinomialNB di solito va meglio: i valori sono "conteggi",
# non veri continui distribuiti come gaussiane.

Esercizio 4 — Train/test split

Stesso dataset cifre, ma ora separa il 25% dei dati per il test e confronta accuracy training vs accuracy test. Cosa noti? 

# Tua soluzione qui

from sklearn.datasets import load_digits
from sklearn.naive_bayes import GaussianNB
from sklearn.model_selection import train_test_split

X, y = load_digits(return_X_y=True)
X_tr, X_te, y_tr, y_te = train_test_split(X, y, test_size=0.25, random_state=42)

m = GaussianNB().fit(X_tr, y_tr)
print(f"Accuracy training: {(m.predict(X_tr) == y_tr).mean():.3f}")
print(f"Accuracy test    : {(m.predict(X_te) == y_te).mean():.3f}")
# L'accuracy di test è la stima onesta della performance su dati nuovi.

Riepilogo

  • Bayes: \(\mathrm P(A\mid B) = \mathrm P(B\mid A)\mathrm P(A)/\mathrm P(B)\), permette di invertire il condizionamento.
  • Si stimano \(\mathrm P(Y)\) e \(\mathrm P(X_i\mid Y)\) come frequenze relative sui dati.
  • L'ipotesi naive \(\mathrm P(X_1, \dots, X_n \mid Y) \approx \prod_i \mathrm P(X_i \mid Y)\) rende l'algoritmo praticabile.
  • La regola di decisione è \(\arg\max_k \mathrm P(Y=y_k) \prod_i \mathrm P(X_i\mid Y=y_k)\).
  • sklearn.naive_bayes offre GaussianNB, MultinomialNB, BernoulliNB, CategoricalNB.
  • Valutare su dati nuovi; con classi sbilanciate guardare oltre l'accuracy.
